## What this is

**Arrow Security** — a multi-tenant security guard operations platform. One company (Arrow Security) is currently active. White-labelling for additional companies comes later.

Three apps ship from this repo:
- **Operations Portal** (`apps/tenant`) — Next.js 16 web dashboard for tenant admins and supervisors
- **Guard App** (`apps/mobile`) — Ionic/Capacitor PWA for guards in the field
- **API** (`apps/api`) — Fastify 4 REST + SSE backend, shared by both frontends

## Authentication & multi-tenancy

**Flow:** `POST /api/auth/login` accepts `{ email, password, tenantSlug }`. The slug resolves to a `tenantId`, which is embedded in the JWT alongside `sub` (userId) and `role`.

**JWT payload:** `{ sub, tenantId, role, iat, exp }` — 24h expiry, signed with `JWT_SECRET`.

**Roles (hierarchy):** `platform_admin > tenant_admin > supervisor > guard > client_viewer`

**Tenant isolation:** Every protected route extracts `payload.tenantId` from the JWT and filters all DB queries by it. Guards additionally have their `guardId` scoped to `payload.sub` on read endpoints.

**Password hashing:** SHA-256 + `PASSWORD_SALT` env var. Not bcrypt — this is a known limitation.

**Auth middleware:** `requireAuth` in `apps/api/src/lib/auth.ts` calls `request.jwtVerify()`. Role-specific helpers: `requireTenantAdmin`, `requireSupervisor`.

## Database schema

All tables in `packages/db/src/schema/`. IDs are `text` primary keys generated by `createId()` (12 random bytes, base64url encoded).

| Table               | Purpose                                              |
|---------------------|------------------------------------------------------|
| `tenants`           | Arrow Security org record (one active row)           |
| `users`             | All users — guards, supervisors, admins, clients     |
| `clients`           | End-client companies (who Arrow guards protect)      |
| `sites`             | Physical locations with lat/lng + geofence radius    |
| `shifts`            | Scheduled guard shifts with status lifecycle         |
| `attendance_records`| Check-in/out events with GPS + method                |
| `patrols`           | Patrol sessions (started → completed)                |
| `checkpoints`       | Named scan points with QR code + optional NFC tag ID |
| `patrol_scans`      | Individual checkpoint scans within a patrol          |
| `incidents`         | Reported incidents with severity + SLA deadline      |
| `guard_locations`   | GPS pings posted every 30s during active shifts      |
| `cameras`           | Camera records (Frigate integration stub)            |
| `payroll_periods`   | Pay period definitions (draft → finalized)           |
| `payroll_records`   | Per-guard pay calculations with ESI/PF (Indian law)  |

Payroll amounts are stored in **paise** (1/100th of a rupee) as integers.

## Live guard location (SSE)

Guards POST `{ latitude, longitude, accuracy, shiftId }` to `POST /api/locations` every 30 seconds while an active shift is running. The API broadcasts each ping to all open SSE connections for that tenant via an **in-memory `Map<tenantId, Set<sendFn>>`**.

The Operations Portal `/map` page connects to `GET /api/locations/live` using `fetch()` with a `ReadableStream` (not `EventSource`) so it can include the `Authorization: Bearer` header. MapLibre GL renders the live map with OSM raster tiles (no API key needed).

Clicking a guard on the map fetches `GET /api/locations/history?guardId=&since=` and draws a coloured polyline trail for the last 8 hours.

**Caveat:** The SSE fan-out is in-process memory only. On multi-server deployments, replace with Redis Pub/Sub.

## Key patterns to follow

**Adding a new API route:**
1. Create `apps/api/src/routes/yourroute.ts` — export `yourRoutes: FastifyPluginAsync`
2. Add `preHandler: requireAuth` on every handler
3. Validate body with Zod inline
4. Scope all DB queries with `eq(table.tenantId, payload.tenantId)`
5. Register in `apps/api/src/server.ts`

**Adding a new DB table:**
1. Create `packages/db/src/schema/yourtable.ts` using `text` IDs with `.$defaultFn(createId)`
2. Export from `packages/db/src/schema/index.ts`
3. Apply to DB: `docker exec securityapp-postgres-1 psql -U secureops -d secureops -c "CREATE TABLE ..."`
   (or run `pnpm push` from `packages/db/` interactively)

**Adding an Operations Portal page:**
1. Create `apps/tenant/src/app/yourpage/page.tsx` with `'use client'`
2. Check `localStorage.getItem('td_token')` in `useEffect`, redirect if missing
3. Add to `NAV` array in `apps/tenant/src/components/Sidebar.tsx`
4. Add API method to `apps/tenant/src/lib/api.ts`

## Scale architecture (build for 500, ceiling at 100k)

The platform must reach 100,000 concurrent users without rewrites. These rules apply to every feature added:

**Non-negotiable from day one:**
- SSE fan-out uses **Redis Pub/Sub** — never an in-process `Map<tenantId, Set<sendFn>>`. The in-process approach already exists but must be replaced before any multi-container deployment.
- All async heavy work goes through **BullMQ** (already Redis-backed): payroll calculation, PDF generation, schedule solving, SMS/email dispatch, OR-Tools jobs.
- **fastify-rate-limit** with Redis store (not in-memory) — in-memory rate limits don't work behind a load balancer.
- **PgBouncer** in transaction mode in front of PostgreSQL — point `DATABASE_URL` at PgBouncer's port.
- **JWT-only auth** — no server-side sessions. Each request is self-contained.
- All DB queries must include `WHERE tenant_id = payload.tenantId`.

**guard_locations schema — partition from MVP:**
```sql
CREATE TABLE guard_locations (...) PARTITION BY RANGE (created_at);
-- Auto-create monthly partitions via pg_partman or cron
```
Add partial GiST index for live-map queries:
```sql
CREATE INDEX guard_locations_geom_recent ON guard_locations USING GIST (location)
  WHERE created_at > now() - interval '8 hours';
```
Keep schema compatible with TimescaleDB (no TimescaleDB-specific types) — upgrade path at 50k+ guards.

**Scale thresholds summary:**
| Users | Key change |
|---|---|
| ≤ 500 | Docker Compose, Redis Pub/Sub SSE, PgBouncer, BullMQ, monthly partitions |
| 5k | Read replica for dashboards, 2–4 Fastify containers behind Traefik |
| 50k | EMQX MQTT replaces HTTP location pings; TimescaleDB on guard_locations |
| 100k | Kubernetes HPA, Citus sharding, PostgreSQL RLS, EMQX cluster |

Full detail in memory file `project_scale.md`.
